# Commit Message Rule

## Общий порядок действий перед коммитом

Выполнять строго последовательно.

### Phase A: Code & State Analysis

1. Выполнить `git status`, чтобы обнаружить любые изменения (включая ручные).
2. Проверить каждый изменённый файл через `git diff`, чтобы определить характер изменений.
3. Сформировать список всех изменений для дальнейших фаз.

### Phase B: Version Bump

1. Оценить, требуют ли изменения (из Phase A) повышения версии.
2. Если да — обновить поле `version` в `package.json` по **SemVer**.

### Phase C: Documentation Sync

1. Убедиться, что файлы в `./docs/` отражают финальное состояние сессии.
2. Убедиться, что `./README.md` отражает текущее состояние проекта; обновить статус при необходимости.
3. **Обновить историю** на основе изменений, выявленных в Phase A, с уже актуальной версией из Phase B:
   - Добавить краткое описание работы сессии в `./docs/Changelog.md` (создать файл, если не существует), **на английском**, максимально **лаконично**.
   - Формат: `## [package.json[version]] YYYY-MM-DD HH:MM\n- Change 1\n- Change 2`.  
   - Если задача полностью завершена — очистить или сбросить `docs/todo.md` для следующего этапа.  

### Phase D: Code Quality Assurance

1. Проверить код на соответствие правилам форматирования TypeScript.
2. Запустить линтер (`npm run lint:fix` / `npm run lint`).
3. Применить автофиксы. Если остались TS-ошибки или lint-проблемы — исправить вручную. **Не игнорировать предупреждения.** Не коммитить с битыми типами или ошибками в документации.

### Phase E: Git Commit

1. **Staging:** `git add` всех изменённых файлов кода (включая lint-фиксы), обновлённых `package.json`, `README.md` и директории `./docs/`.
2. **Commit Message:** Генерировать семантическое сообщение коммита (`type(scope): subject`).
   - Указывать изменения в TS / инфраструктуре, если применимо.
   - Использовать `chore: bump version`, если обновлялась версия.
3. **Execution:** Выполнить коммит.
