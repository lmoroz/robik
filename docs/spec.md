# Спецификация

## 1. Цель

Разработать Telegram-бота, который работает как простой AI-чат поверх локальной LLM и отвечает на входящие текстовые сообщения без хранения истории диалога.

## 2. Источник требований

Исходная постановка задачи: [Yonote](https://larchanka.yonote.ru/share/0651382b-e62a-4c82-8da4-d13315ae758b/doc/telegram-bot-s-lokalnoj-llm-NLQbGlDRTJ).

## 3. Целевой результат

На выходе должен быть минимальный, но рабочий проект, который:

- подключается к Telegram Bot API через `polling`;
- отправляет пользовательский текст в локальную LLM;
- возвращает сгенерированный ответ в тот же чат;
- не хранит историю сообщений;
- не использует БД;
- не зависит от облачных LLM.

## 4. Пользовательские сценарии

### Сценарий 1. Обычный текстовый запрос

1. Пользователь отправляет боту текстовое сообщение.
2. Бот получает текст через Telegram polling.
3. Бот отправляет текст в локальную LLM.
4. Бот получает ответ модели.
5. Бот возвращает ответ в Telegram.

Ожидаемый результат: пользователь получает текстовый ответ без заметной внутренней сложности.

### Сценарий 2. LLM недоступна

1. Пользователь отправляет сообщение.
2. Бот пытается обратиться к локальной LLM.
3. LLM не отвечает или возвращает ошибку.
4. Бот отправляет пользователю понятное сообщение об ошибке.

Ожидаемый результат: процесс не падает, бот продолжает работать для следующих запросов.

### Сценарий 3. Несколько запросов подряд

1. Пользователь отправляет несколько сообщений одно за другим.
2. Бот обрабатывает их последовательно и стабильно.

Ожидаемый результат: бот не зависает, не требует перезапуска, не смешивает ответы.

## 5. Функциональные требования

- Бот обязан отвечать на любые текстовые сообщения.
- Бот обязан передавать текст в локальную LLM.
- Бот обязан отправлять ответ обратно в Telegram.
- Бот обязан корректно обрабатывать ошибку недоступности LLM.
- Бот должен стабильно работать при серии запросов подряд.

## 6. Нефункциональные требования

- Архитектура должна быть простой и читаемой.
- Реализация должна быть минимальной по количеству сущностей и зависимостей.
- Код должен быть понятен студенту и легко объясняться на защите.
- Конфигурация должна задаваться через переменные окружения.

## 7. Ограничения

- Только локальная LLM.
- Только `polling`.
- Без `webhook`.
- Без хранения истории диалога.
- Без базы данных.
- Без облачных LLM в runtime.
- Код должен быть сгенерирован с помощью LLM полностью или по частям с последующей доработкой.

## 8. Не-цели

В этот проект не входят:

- память диалога;
- пользовательские профили;
- хранение логов в БД;
- панель администратора;
- RAG;
- инструменты, агенты, function calling;
- multi-model orchestration;
- сложная маршрутизация запросов.

## 9. Выбранная техническая стратегия

- Язык: `TypeScript`
- Runtime: `Node.js`
- Telegram transport: библиотека с поддержкой `polling`
- Local LLM runtime: `Ollama`
- Модель по умолчанию: `qwen3:0.6b`

Причина выбора: это самый короткий путь к рабочему результату без лишней инфраструктуры.

## 10. Архитектура

```text
Telegram Update
  -> Bot Handler
  -> LLM Client
  -> Bot Response
  -> Telegram API
```

### Основные модули

- `config` — загрузка и валидация переменных окружения;
- `telegram` — инициализация бота и обработка update;
- `llm` — клиент для вызова Ollama;
- `handlers` — логика ответа на текстовые сообщения;
- `app` — точка входа и запуск polling.

## 11. Контракт обработки сообщения

Вход:

- `chatId`
- текст сообщения

Обработка:

- проверить, что сообщение текстовое;
- передать текст в локальную LLM;
- получить строковый ответ;
- вернуть ответ пользователю.

Выход:

- одно текстовое сообщение в Telegram.

## 12. Обработка ошибок

- Если сообщение не текстовое, бот может мягко игнорировать его или вернуть короткое пояснение.
- Если LLM недоступна, бот должен вернуть понятное сообщение вида:
  `Локальная модель сейчас недоступна. Попробуй ещё раз позже.`
- Если Telegram API вернул ошибку отправки, ошибка должна быть залогирована, а процесс не должен завершаться.

## 13. Бонусные возможности

Если останется время, можно добавить:

- `/start`
- `SYSTEM_PROMPT`
- переключение модели через команду
- базовое логирование запросов

Это не должно ломать основную простую архитектуру.

## 14. Критерии приёмки

- Бот запускается локально.
- Telegram-сообщение приводит к вызову локальной LLM.
- Ответ LLM возвращается в Telegram.
- История диалога не хранится.
- Без базы данных.
- Без webhook.
- Есть инструкция по запуску.
- Есть краткое описание, какую LLM использовали и как код генерировался с помощью LLM.
